# create-rd-todo — NotionのR&D TODOチケット作成・更新

ユーザーから依頼された内容を、R&D TODOリスト(Notionデータベース)の固定テンプレート・書き振りに整形して、新規作成(`notion-create-pages`)または既存チケットの補完(`notion-update-page`)を行う。GitHub版の `create-issue` のNotion R&D版。

作成・更新で**書き振り・用語基準・「既定で目的とゴールのみ記入」の方針は完全に共通**。新規は「## 手順(作成時)」、既存補完は「## 手順(更新・補完時)」を参照。

## 投入先(固定)

- データベース: **🐙 R&D TODOリスト**
- 親指定: `parent = { type: "data_source_id", data_source_id: "2deda318-eeda-8045-b670-000b542f04d9" }`
- **注意**: このDBには既定テンプレートがあるが、MCPの `create-pages` では自動適用されない。だから本文テンプレートを必ずこのスキル側から焼き込む(空ページにしない)。

## 手順(作成時)

1. **コードを確認してから書く**: 実装・検証に関わるチケットは、関係するファイル・関数・設定を実際に読み、具体的なパス付きで書く(憶測で書かない)。確認できなかった点は「要確認」と明示する。
2. **本文を組み立てる**: 下記テンプレートに沿う。**既定では「目的」と「ゴール」だけ記入し、「報告」「優先度判定」は空欄(定型プレースホルダ)のまま残す**(報告は着手後に追記、優先度判定は起票者が手で採点する運用)。ユーザーが明示的に報告・優先度も書けと言った場合のみ埋める。
3. **プロパティを決める**: 名前(必須)・ラベル(文脈推定)・ステータス(既定 `Backlog`)。担当者・締切は指示があれば。
4. `notion-create-pages` で作成し、**作成されたページURLを報告**する。
5. 作成後、目的・ゴールの書き振りに改善案があれば提案する(ユーザーが望めば `update-page` で差し替え)。

## 手順(更新・補完時)

既存チケット(NotionチケットURL指定)を補完する場合:

1. **現状を取得**: `notion-fetch` でページを取得し、薄い節・プレースホルダ(`最終ゴール`・`hoge` 等)・既に埋まっている節を把握する。
2. **根拠を集める**: 本文中の関連ページ(検証結果・関連チケット等)を `notion-fetch` で辿り、実装に関わるならコードも読む(作成時と同じ「憶測で書かない」原則)。具体的な数値・症状・パスで裏付ける。
3. **目的・ゴールを厚くする**: 下記「目的・ゴールの書き振り」に沿って補完する。**既定では目的・ゴールのみ**を対象にし、「報告」「優先度判定」は触らない(ユーザーが明示した場合のみ)。
4. **`update_content` で部分置換**: ページ全体を置き換える `replace_content` は使わず、`notion-update-page` の `update_content`(old_str/new_str)で目的・ゴールの該当箇所だけを差し替える。他節・子ページ・既存の関連ページmentionを壊さない。
5. **プレースホルダは消さず指摘**: `hoge` 等の起票者メモらしき残骸は勝手に削除せず、補完後に「残っているが消すか?」と確認する。
6. 変更点と、要確認(コードが見つからない等)・残ったプレースホルダを報告する。

## 目的・ゴールの書き振り

- **目的**: 「なぜ必要か(背景)」+「探索の問い/やること」。探索系タスクなら、検証したい仮説(有効性・効率・頑健性など)を箇条書きの主役に出す。実装系なら、やることを列挙。
- **ゴール**: プロセス列挙ではなく**成果物・到達状態・判断**で書く(再現可能なコード/結果まとめ/Go-No-Go判断と根拠など)。チェックリスト形式 `- [ ]`。成功基準があれば判断項目に紐づける。
- **用語レベル**: 一般的な技術カタカナ語(ベースライン/コールドスタート/混同行列/F1/有意/頑健性 等)はそのまま残す。社内造語・非自明な語(エンロール→声紋に登録、アブレーション→条件を変えた比較、poisoning→誤修正で声紋が汚れる、eviction→悪い声紋を外す 等)のみ言い換える。全部平易化する「やりすぎ」は避ける。迷う語は1回だけ簡潔に補足を添える。

## 生の声(お客様コメント・営業メモ・ユーザーフィードバック)の扱い

チケットには顧客や営業の生の声が含まれることがある(多くは「目的」欄の中)。これは一次情報で説得力の核なので、作成・更新を通じて**最優先で保護**する。

- **verbatim 保持**: 要約・言い換え・削除をしない。引用ブロック(コードブロックや quote)のまま、文言・改行をそのまま残す。
- **出所・文脈を保つ**: 誰の声か・日付・前提条件(例「AIカスタマイズで声紋を各人3件登録した状態でこの精度」)を落とさない。
- **生の声と起票者(自分)の記述を視覚的に分離・帰属させる**: 後から別の人が読んで「どこが顧客/営業の生の声で、どこがAI/起票者が整理した内容か」を一目で判別できる構造にする。
	- 生の声は引用(quote)またはコードブロックに入れ、**直前に出所ラベル**を付ける(例「お客様の声(2026-04-10)」「営業メモ(担当: ◯◯)」)。複数あればそれぞれラベルで区別。
	- 自分が書いた背景・分析・探索の問いは、引用の外に地の文/箇条書きで置く。**生の声を自分の地の文に言い換えて混ぜ、出所を曖昧にしない**(参照するなら引用する)。必要なら「(起票者整理)」等で自分の記述だと分かるようにする。
- **「目的・ゴールのみ編集可」でも生の声は編集対象外**: 目的の中に生の声がある場合、手を入れてよいのは自分が書いた地の文だけ。生の声は引用のまま残し、背景・問いはその引用を**根拠として**組み立てる。整理のため引用を別ブロックへ移動するのは可だが、文言の編集は不可。
- 要約・削除したい合理的理由があるときは、勝手に行わず確認する。

## 本文テンプレート(Notion-flavored Markdown)

callout・チェックリスト・コードブロックの記法は MCPリソース `notion://docs/enhanced-markdown-spec` に従う(callout子要素はタブ字下げ)。3つの💡calloutの文言は下記のまま固定で出す。

````markdown
# 目的
<callout icon="💡" color="gray_bg">
	なぜこのタスクが必要となったか<br>探索系タスクなら知りたいこと、実装系タスクならやることは何か
</callout>
- (背景: なぜ必要か)
- (探索の問い or やること。探索系は仮説を入れ子の箇条書きで)
	- **観点A**: ...
	- **観点B**: ...
# ゴール
<callout icon="💡" color="gray_bg">
	タスク完了時に到達して欲しい状態<br>出して欲しいアウトプット（実験結果まとめ、再現可能なコード、判断材料など）
</callout>
- [ ] (成果物・到達状態1)
- [ ] (成果物・到達状態2 / 判断と根拠。成功基準を紐づける)
# 報告
<callout icon="💡" color="gray_bg">
	検討履歴が分かるように報告日ごとに記載<br>最終的な結論が何か明確に分かるように記載
</callout>
# 優先度判定（起票者利用）
```plain text
【総合点】

Reach * Impact * Confidence / Effort
◯点

【個別評価】

Reach／リーチ：影響を受けるユーザー数
全ユーザーor主要KPIに直結=5／多くのユーザー（過半数）=4／一部のユーザー（特定セグメント）=3／ごく一部（限定条件）=2／内部利用・例外ケース=1
◯点

Impact／影響：各ユーザーにどの程度影響するか
影響大=3／影響中=2／影響小=1
◯点

Confidence／信頼性：リーチ、影響、労力の見積の自信の度合い
信頼度高=100%／信頼度中=80%／信頼度低=50%
◯%

Effort／労力：実装するのにかかる時間
労力小＝2(~1週間)／労力中＝3(1~2週間)／労力大＝5(3週間~)
◯点
```
````

## プロパティ仕様

`create-pages` の `properties` に渡す(プロパティ値は文字列/数値/null):

- **名前**(title, 必須): 簡潔な日本語。分類用に先頭へ `[ラベル]` を付けてよい(例: `[話者認識] ...の検証`)
- **ステータス**: 次のいずれか — `Backlog` / `ToDo` / `Doing` / `Review` / `Done` / `Archive`。既定は `Backlog`
- **ラベル**(multi-select): JSON配列文字列で渡す(例: `"[\"話者認識\"]"`)。選択肢: `プライベートAI` / `音声認識` / `新機能` / `話者認識` / `Ops` / `翻訳` / `リアルタイム` / `校正` / `要約` / `その他` / `SMC` / `SM`。文脈から該当するものを推定し、不確実なら確認する
  - **`新機能` の意味に注意**: 「新規開発」という意味ではなく、**既存の製品機能カテゴリ(`音声認識` / `話者認識` / `要約` / `校正` / `翻訳` / `リアルタイム` 等)のいずれにも当てはまらない領域**に付けるラベル。該当する機能カテゴリがあるならそちらを使い、`新機能` は付けない(例: 話者認識まわりのチケットは `話者認識` のみで、`新機能` は不要)
- **担当者**(person, 任意): 指定された場合のみ。`notion-get-users` で名前→ユーザーIDを解決し、JSON配列文字列で渡す
- **締切日**(date, 任意): 展開形式 `"date:締切日:start"`(ISO日付), 必要なら `"date:締切日:end"` / `"date:締切日:is_datetime"`
- **優先度**(number, 任意): 0〜8。優先度判定を埋める運用のときのみ、RICEの `Reach*Impact*Confidence/Effort` を計算して数値でセット。既定は未設定

## 運用ルール

- 言語は日本語
- 1チケット = 1関心事。独立した要望が複数混ざっていたら分割を提案
- 既定では報告・優先度判定は空欄のまま(上記手順2)
- 作成後はURLと、設定したプロパティ(ステータス・ラベル等)を報告する
