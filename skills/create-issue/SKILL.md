# create-issue — 統一テンプレートでのGitHub issue作成

ユーザーから依頼された内容を、以下の固定テンプレートに整形して `gh issue create` で作成する。

## 手順

1. **対象リポジトリの特定**: 明示されなければカレントの git remote から推定し、作成前にリポジトリ名を確認に出す(別リポジトリへの誤爆防止)
2. **コードを確認してから書く**: 実装に関わるissueは、関係するファイル・関数・設定を実際に読み、実装ポイントを具体的なパス付きで書く(憶測で書かない)。確認できなかった点は「要確認」と明示する
3. テンプレートに沿って本文を組み立て、`gh issue create --repo <owner/repo> --title "..." --body "..."` で作成
4. 作成されたissue URLを報告する

## タイトル規約

- 日本語・簡潔(50字以内目安)。「〜する」「〜を追加する」など動作が分かる形
- プレフィックス不要(ラベルで分類する)。バグの場合のみ「[bug]」を付けてよい

## 本文テンプレート

```markdown
## 背景・目的

なぜ必要か。現状の問題・きっかけとなった事実を書く。
**実際に起きた事例・データ・関連ドキュメントへの参照があれば必ず引用する**(説得力の核)。

## 提案

何をするか。実装に関わる場合は、変更が必要な箇所を順序立てて列挙:

1. **`path/to/file`**: 具体的な変更内容
2. ...

設計判断の理由(例: トークン増なし、後方互換など)があれば添える。

## 受け入れ条件

- [ ] 検証可能な完了条件(「〜が含まれる」「〜でも保持される」)
- [ ] エッジケース・既存データの扱い方針の決定事項
- [ ] テスト追加(該当する場合)

## 補足

(任意)関連issue・スコープ外事項・周辺システムへの影響など
```

## 運用ルール

- 言語は日本語(リポジトリの既存issueが英語ならそれに合わせる)
- 1 issue = 1 関心事。複数の独立した要望が混ざっていたら分割を提案する
- ラベル・アサイン・マイルストーンは指示があれば `--label` 等で付与
- バグ報告の場合は「背景・目的」を「現象 / 再現手順 / 期待動作」に置き換えてよい(それ以外の節は維持)
