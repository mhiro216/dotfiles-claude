# SNS投稿文の一括作成（X公式 / X個人 / Facebook）

指定された記事URLから、3チャネル分の投稿文を一度に生成する。

## 入力

- 引数として記事URLが渡される場合：そのURLを使う。
- URLが2つ渡される場合：対になる記事（例：Zenn=How編とnote=Why編）として扱い、相互に言及・併記してよい。
- 引数が空の場合：ユーザーに記事URLを尋ねてから進める。

## 手順

1. WebFetch で記事を取得し、タイトル・著者・主なトピック・要点（数値・非自明な知見）・対象読者を把握する。
2. 下記のチャネル別ルールに従って3本の投稿文を作成する。
3. X向け2本はX換算の文字数を見積もり、280以内（目標270前後）に収まっているか確認する。超える場合は圧縮して作り直す。
4. 指定の体裁で一括出力する。

## チャネル別ルール

### 1. X 会社公式（@Nishika0507）

企業としての技術情報発信。想定読者はAIエンジニア。

- トーン：落ち着いた、誠実な企業発信。絵文字・過度な感嘆符なし。ハッシュタグ原則なし。
- 構成：
  - 1行目：媒体に合わせた定型文。Zenn記事なら `Nishika技術ブログを更新しました。`、note記事なら `Nishikaのnoteを更新しました。`
  - 本文：記事の最も非自明で興味深い知見を中心に1〜2文。具体的な数値（データ規模・スコア等）があれば入れる。
  - 末尾：記事URL。
- 記事タイトルの文言はリンクカードに出るため、投稿文では繰り返さない。

### 2. X 個人（@mhiro216 / 松田裕之）

個人アカウント。普段の投稿は一人称の業界コメント・意見表明型。フォロワーは技術・ビジネス混在。

- 定型の告知文（「〜を更新しました」）は使わない。**本人の気づき・意見を地の文で語り、記事は補足として置く**形にする。
- 一人称（私）視点。「やってみて気づいたのは〜」「〜だと思っている」のような観察・主張先行。
- 対になる記事が2本ある場合は、1ポストに `How:` / `Why:` のように併記して回遊を作ってよい。
- 絵文字・ハッシュタグなし。記事タイトルの転載ではなく、タイトルの核心フレーズを本人の言葉として地の文に溶かす。

### 3. Facebook 個人

つながりはビジネス系（経営者・事業責任者層）中心。

- **告知ではなく「近況＋学びの共有」**として書く。4〜6段落の長文。冒頭に軽い自虐・時事フックを入れてよい。
- **本文だけで価値が完結させる**（リンクを踏まない読者が大半。外部リンク付き投稿はリーチも絞られる）。リンクは「詳しくはこちら」として末尾に置く。
- 記事の主張を**「これは自社固有の話ではなく、どの会社にもある話」へ一般化**する段落を必ず入れ、読者が自分ごと化できるようにする。
- ツール固有名詞（Claude Code等）はフック（冒頭・具体描写）に使ってよいが、一般化のテーゼ部分では「AI」に戻す（主張がツール宣伝に見えないように）。
- 任意：コメントを誘う問いかけを1文（「皆さんの会社の『宙に浮いている仕事』は何ですか」等）。エンゲージメントがリーチを広げる媒体のため有効。
- 採用募集の直接的な文言は入れない（記事側で回収させる）。
- 末尾にもう一方の記事（エンジニア向け）への誘導を一行添えてよい。

## 共通の文体ルール

- 日本語。専門用語はエンジニアに通じる標準的な表記を使う。
- 慣用句は正しい形で使う（例：「ご多分に漏れず」。「多分に漏れず」としない）。
- 「思うのは〜と思います」のような文のねじれ・同義反復がないか出力前に見直す。
- 表記揺れ（「一度/1度」等）を統一する。

## 文字数（X向け2本のみ）

- Xの上限280に必ず収める（全角=2、半角=1、URLは23でカウント）。
- 余裕をもって270前後を目標にする。
- 超える場合は、要点（数値・主要な構成・非自明な知見）を残して圧縮する。
- Facebookに文字数制限はないが、4〜6段落・800字程度を上限の目安にする。

## 出力

以下の順で、それぞれ見出しを付けてコードブロックで提示する：

1. **X 会社公式（@Nishika0507）** ＋ 概算文字数（X換算）
2. **X 個人（@mhiro216）** ＋ 概算文字数（X換算）
3. **Facebook 個人**

最後に、チャネル間で表現を変えた意図・主な編集ポイントを箇条書きで簡潔に説明する。あわせて投稿運用の定石を一行ずつ添える：公式と個人で同文を貼らない／個人ポストが伸びたら公式がRT／連投せず数時間〜1日空ける。
